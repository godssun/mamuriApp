## Context

프리미엄 쿼터 제한 도입 시, 위기 상황(자해/자살 키워드 감지)의 사용자가 페이월에 차단되는 상황은 절대 발생해서는 안 된다. 위기 감지 시점, 플래그 지속 기간, 키워드 범위를 결정해야 한다.

---

## Decision

**위기 유저는 쿼터와 무관하게 항상 AI 응답을 받는다.**

구현 방식:
1. SafetyCheck를 QuotaCheck보다 항상 먼저 실행
2. 위기 감지 시 SafetyEvent 기록 + 7일 플래그 설정
3. 7일 이내 재발 시 플래그 자동 연장
4. 위기 플래그 활성 시 쿼터 무시, 페이월 미표시
5. 위기 키워드를 13개→25개로 확장 (간접/완곡 표현 포함)
6. 모든 위기 응답에 긴급 연락처(1393, 1577-0199) 포함

---

## Alternatives Considered

| 대안 | 장점 | 단점 | 기각 사유 |
|------|------|------|-----------|
| 위기 감지 없이 전체 무료 | 안전 보장 | 비즈니스 모델 파괴 | 지속 불가능 |
| 72시간 플래그 | 급성 위기에 적합 | 아급성 위기 미보호 | 보호 기간 부족 |
| 14일 플래그 | 더 안전 | 오탐 시 장기 무료 사용 | MVP에서 7일로 시작, 데이터 수집 후 조정 |
| AI 기반 위기 감지만 | 높은 정확도 가능 | 추가 API 비용, 지연 | Phase 3에서 도입 |

---

## Trade-offs

- **선택**: 키워드 기반 + 7일 플래그 + 자동 연장
- **장점**: 구현 단순, 즉각 보호, 비용 예측 가능
- **단점**: False Positive 존재 (과장된 표현도 감지), 간접 표현 사각지대
- **수용 근거**: False Positive은 안전 측면에서 수용 가능 (과보호 > 미보호)
- **향후 계획**: Phase 3에서 AI 기반 감지로 정확도 향상

---

## Long-Term Impact

- 위기 유저 데이터는 수익화 결정에 절대 사용 금지 (윤리 원칙)
- 3단계 출시: 소프트 론칭 → 제한적 시행 → 전면 시행 (위기 감지 정확도 >95% 달성 후)
- 심리학 전문가 자문 필요 (3개월마다)
- PIPA(개인정보보호법) 준수: SafetyEvent는 민감정보로 별도 동의 필요
